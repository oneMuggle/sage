## 2. 接口契约

### 2.1 SLA 指标定义

| 指标 | 定义 | 测量方式 |
|------|------|----------|
| 延迟 | 请求从接收到响应完成的时间 | 服务端打点 |
| 吞吐量 | 单位时间内处理的请求数 | 计数器/时间窗口 |
| 错误率 | 失败请求占总请求的比例 | 错误计数/总计数 |
| 可用性 | 系统正常响应的时间占比 | 健康检查成功率 |
| 内存使用 | 进程 RSS 内存 | 系统指标 |
| CPU 使用 | 进程 CPU 占用率 | 系统指标 |

### 2.2 SLA 目标

| 指标 | 目标 | P50 | P95 | P99 |
|------|------|-----|-----|-----|
| API 延迟（简单 CRUD） | < 200ms | < 50ms | < 200ms | < 500ms |
| API 延迟（复杂查询） | < 1000ms | < 200ms | < 1000ms | < 3000ms |
| 数据库查询 | < 50ms | < 10ms | < 50ms | < 200ms |
| IPC 消息传递 | < 50ms | < 10ms | < 50ms | < 100ms |
| 吞吐量 | > 100 req/s | - | - | - |
| 错误率 | < 1% | - | - | - |
| 可用性 | > 99.5% | - | - | - |

### 2.3 资源限制

| 资源 | 软限制 | 硬限制 | 备注 |
|------|--------|--------|------|
| 后端内存 | 512MB | 1GB | 超出后触发 GC + 告警 |
| 后端 CPU | 50% | 80% | 单核基准 |
| SQLite 连接数 | 10 | 50 | 连接池配置 |
| 数据库大小 | 500MB | 2GB | 超出后归档旧数据 |
| 日志大小 | 100MB | 500MB | 自动轮转 |

## 3. 不变量约束

### 3.1 数据不变量

#### 不变量 1: P95 延迟达标

**定义**：在正常负载下（< 80% 资源使用），API P95 延迟不超过 SLA 目标。

**验证方法**：
```python
async def test_p95_latency():
    latencies = []
    for _ in range(1000):
        start = time.perf_counter()
        await api_call("/api/sessions")
        latencies.append((time.perf_counter() - start) * 1000)
    p95 = numpy.percentile(latencies, 95)
    assert p95 < 200, f"P95 latency {p95:.1f}ms exceeds 200ms SLA"
```

**检查频率**：
- [x] 每次性能测试（CI nightly）
- [ ] 每小时（生产监控）

#### 不变量 2: 错误率 < 1%

**定义**：在正常负载下，API 错误率（5xx）不超过 1%。

**检查频率**：每分钟滚动窗口（生产监控）、每次性能测试

#### 不变量 3: 内存不泄漏

**定义**：在持续负载下，内存使用量不持续增长。1 小时内内存增长 < 10%。

### 3.2 行为不变量

#### 负载下稳定性

**定义**：在 2x 正常负载下持续运行 30 分钟，错误率 < 5%，无 OOM。

#### 优雅降级

**定义**：超过硬限制时，系统返回 503 而非崩溃。

### 3.3 性能回归检测

**定义**：每次 PR 的性能测试结果与基线对比，回归 > 10% 则阻断合并。

## 4. 失败模式与恢复

### 4.1 失败模式 1: 内存泄漏

**触发条件**：未关闭的数据库连接、缓存无上限增长、事件监听器未注销

**检测方式**：内存使用持续上升，GC 后不回落

**恢复策略**：自动重启进程（临时），分析 heap dump 定位泄漏源，修复代码 + 回归测试

### 4.2 失败模式 2: 数据库慢查询

**触发条件**：缺少索引、数据量超预期增长、查询计划变更

**检测方式**：慢查询日志（> 100ms）

**恢复策略**：添加/优化索引，重写查询，归档旧数据减少表大小

### 4.3 失败模式 3: 吞吐量瓶颈

**触发条件**：单线程阻塞、连接池耗尽、外部依赖延迟

**恢复策略**：水平扩展（多 worker），增大连接池，添加缓存层

### 5.1 负载测试

**位置**：`tests/performance/load/`

**位置**：`tests/performance/stress/`

**运行命令**：
```bash
/home/fz/anaconda3/envs/sage-backend/bin/pytest tests/performance/stress/ -v
```

**验证**：内存增长 < 10%、错误率 < 0.1%、延迟无持续上升

## 6. 监控指标

### 6.1 运行时指标

| 指标 | 类型 | SLA 目标 | 告警阈值 | 监控方式 |
|------|------|----------|----------|----------|
| API P95 延迟 | 直方图 | < 200ms | > 500ms | Prometheus |
| API P99 延迟 | 直方图 | < 500ms | > 1000ms | Prometheus |
| 错误率 (5xx) | 计数器 | < 1% | > 5% | Prometheus |
| 吞吐量 | 计数器 | > 100/s | < 50/s | Prometheus |
| 内存使用 (RSS) | 仪表 | < 512MB | > 800MB | 系统监控 |
| CPU 使用率 | 百分比 | < 50% | > 80% | 系统监控 |
| 数据库查询 P95 | 直方图 | < 50ms | > 200ms | 应用指标 |
| 可用性 | 百分比 | > 99.5% | < 99% | 健康检查 |

### 6.2 告警规则

| 规则 | 条件 | 严重性 | 通知方式 |
|------|------|--------|----------|
| 高延迟 | P95 > 500ms 持续 5min | WARNING | Slack |
| 高错误率 | 5xx > 5% 持续 2min | CRITICAL | Slack + 短信 |
| 内存告警 | RSS > 800MB 持续 5min | WARNING | Slack |
| 服务不可用 | 健康检查连续 3 次失败 | CRITICAL | Slack + 短信 |
| 磁盘空间 | 使用率 > 90% | WARNING | Slack |
